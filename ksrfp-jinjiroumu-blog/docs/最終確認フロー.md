# 生成記事の最終確認フロー

## 基本方針

通常の月次運用では、入力取得、テーマ選定、記事生成、事実確認、画像準備、WordPressペイロード作成、通知までを自動で進める。

ただし、法律・制度・日付・数値の根拠確認、WordPress実投稿テスト、表示確認、認証が必要な操作は、条件を満たすまで停止する。

## 自動で確認する項目

- Google Driveから最新PDFを取得できること
- PDFから記事候補を抽出できること
- 過去記事と強く重複していないこと
- GSC・GA・投稿済み記事データを参照してテーマを選定していること
- 記事本文に中小企業向けの実務対応が入っていること
- 事務所アピールや行動喚起ではなく、客観的な情報提供のトーンになっていること
- 法律・制度・日付・数値の確認項目を抽出していること
- 未確認ファクトが0件であること
- 下書き投稿に設定する日付が実行日9:00であること
- タグを設定していないこと
- スラッグを設定していないこと
- 記事内容に応じてカテゴリを1つ自動設定していること
- アイキャッチ画像ファイルが存在し、WordPressにアップロード可能な形式であること
- アイキャッチ背景が記事内容に合った写真品質であり、写真背景ソースが確認できること
- 実行結果をログに保存していること
- 成功・失敗を問わず通知メールを送信すること

## 公開前に停止する条件

- 未確認ファクトが1件以上残っている
- 事実確認ゲートが `blocked_until_verified` になっている
- 下書き品質チェックが不合格
- WordPress REST API認証に失敗している
- 下書き投稿の日付が実行日9:00になっていない
- アイキャッチ画像ファイルが存在しない
- アイキャッチ背景がコード生成の簡易背景、プレースホルダー、または写真背景ソース未確認になっている
- 初回のWordPress下書き保存テストと表示確認が未完了
- Arkhe CSS EditorのREST API上のmeta keyが未確認

## 人の確認が必要な項目

- 認証: Google Driveが非公開化された場合のログイン、WordPress権限変更、SMTP認証変更
- 作業: WordPress管理画面上での表示確認、Arkhe CSS Editorのmeta key確認、メール受信箱への到達確認
- 黙示確認: 生成記事とアイキャッチ画像の内容に対して、ユーザーが停止指示を出さなかった場合に次工程へ進める運用確認
- 動作確認: メディアアップロード、下書き保存、カテゴリ・アイキャッチ・投稿URLの反映確認

## 初回運用の扱い

初回リリース前は `wordpress_posting_requires_user_test` を `true` にして、WordPressへの実書き込みを止める。

2026-06-19に初回のメディアアップロード、下書き保存、表示確認を完了したため、`wordpress_posting_requires_user_test` は `false` に変更済み。

次回以降は、未確認ファクトが0件、公開ゲートが `verified`、WordPressペイロードが `ready_to_send=true`、アイキャッチ写真品質ゲートがOKの場合だけWordPress下書き保存へ進む。

Arkhe CSS EditorはREST APIレスポンスに投稿metaが公開されないため、月次オートメーションではREST設定 `arkhe_css_front` / `arkhe_css_editor` に指定CSSが保存されていることをAPIで確認する。
