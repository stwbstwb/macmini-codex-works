# 成功実装比較メモ

作成日: 2026-06-20

## 確認した外部実装

GitHub `stwbstwb/geekom-codex-works` は private repository のため、Codex GitHubコネクタでは `404 Not Found` だった。ローカル `gh` 認証では参照できた。

確認対象:

- `westhill-blog-automation`
- `wordpress-article-automation`

## 成功実装の重要設計

### 1. 投稿前プリフライト

`wordpress-article-automation` では、WordPress投稿やメール送信の前に次を同一実行環境で確認している。

- WordPress REST API endpoint のDNS/TCP疎通
- WordPress認証情報の有無
- WordPress REST APIで `context=edit` の読み取りができること
- Gmail SMTP設定の有無
- 必要に応じたSMTPログイン確認

関連ファイル:

- `wordpress-article-automation/shared/automation/preflight_environment.py`
- `wordpress-article-automation/shared/network_probe.py`
- `wordpress-article-automation/shared/wp_connection_check.py`
- `wordpress-article-automation/NETWORK_TROUBLESHOOTING.md`

### 2. 投稿前チェックに失敗したらWordPress投稿へ進まない

`AUTOMATION_GUIDE.md` と `weekly_new_posts_prompt.md` では、環境チェック、品質チェック、preflight のいずれかが失敗した場合、WordPress投稿へ進まないことが明記されている。

今回の案件では、Google Drive確認用テキスト保存後にWordPress投稿でDNS失敗したため、成果物数がずれた。成功実装の考え方に合わせるなら、外部連携プリフライトを最初に行い、WordPress/SMTPが使えない実行環境ではDrive保存やWordPress投稿へ進ませない。

### 3. 投稿後の読み返し検証

成功実装では、WordPress作成後に投稿IDを使ってREST APIから読み返し、実際のWordPress状態を検証している。

確認項目例:

- `draft`
- 日付
- `featured_media`
- 本文内画像
- CTA
- 本文構造

関連ファイル:

- `wordpress-article-automation/shared/inspect_post.py`
- `wordpress-article-automation/shared/automation/report.py`

この案件でも `run_wordpress_verify_batch.py` があるため、投稿成功の定義は「create_post完了」ではなく「3件すべてWordPress読み返し検証OK」とする。

### 4. 通知は最終通知1回に寄せる

`westhill-blog-automation` では、通知メールに `send_notification_once()` があり、runフォルダ内に `notification_sent.json` marker を残す。成功送信済みのrunでは再実行しても同じ通知を送らない。

関連ファイル:

- `westhill-blog-automation/shared/notifications/email.py`

今回の案件では、生成段階通知、手動通知、最終通知が混在し、過去テストで通知過多が起きた。成功実装に合わせ、run単位の通知markerを導入し、最終通知だけを送る。

### 5. 再実行時の二重投稿防止

`westhill-blog-automation` では `RunState` があり、既に `scheduled_post` が記録済みなら再投稿せず終了する。

関連ファイル:

- `westhill-blog-automation/shared/run_state/state.py`
- `westhill-blog-automation/westhill/weekly_new_post.py`

今回の案件では、WordPressタイトル重複ガードは追加済みだが、run単位の「このrunは既にWordPress下書き作成済み」という状態管理が弱い。成功実装に合わせ、run状態で二重作成を防ぐ。

### 6. メール送信の実装は基本的に同系統

成功実装も、この案件も、SMTP送信は `smtplib.SMTP(...).starttls().login().send_message()` であり、方式自体は大きく違わない。

差分は主に次の3点。

- 成功実装は `.env.local` から設定を読み、preflightでSMTPログイン確認を行う。
- 成功実装は送信済みmarkerで重複通知を防ぐ。
- 今回案件はSMTP失敗時にsendmail fallbackを試すが、Codex/OS権限で `postdrop` が失敗するため、本番通知の主経路としてはSMTPを必須扱いにするべき。

## 今回の三回目最終検証の失敗原因

### WordPress下書き保存: NG

直接原因:

- 自動実行中のWordPress REST API接続でDNS/外向き通信エラーが発生した。
- エラーは投稿作成前、同名タイトル重複チェックのWordPress読み取り時に発生した。

構造的原因:

- 外部連携プリフライトが「同じ自動実行環境」で強制ゲートになっていなかった。
- Drive保存がWordPress成功前に実行され、Driveだけ増える状態を許した。

### 通知メール: NG

直接原因:

- SMTPもDNS/外向き通信エラーで失敗した。
- sendmail fallbackも `/usr/sbin/postdrop` 権限エラーで失敗した。

構造的原因:

- SMTPログインの事前確認が最終実行直前の必須ゲートになっていなかった。
- 通知送信済みmarkerがないため、過去テストでは通知過多も起きやすかった。

### 記事本文品質: NG

直接原因:

- 同一記事内に同じ定型段落が20回以上反復した。

構造的原因:

- 既存の品質ゲートは記事間類似度や禁止語を見ていたが、同一記事内の段落反復を検出していなかった。
- 自動実行では、記事生成後に「人間が読めばすぐ分かる不自然さ」を機械ゲートで止めきれていなかった。

## 採用すべき解決方針

### A. 本番同等プリフライトを最初に実行する

自動実行の冒頭で、次を必須チェックする。

- Google Drive PDF一覧取得
- WordPress REST API読み取り
- WordPress認証
- WordPressカテゴリ取得
- SMTP DNS/TCP疎通
- SMTPログイン

このプリフライトが失敗したら、記事生成・Drive保存・WordPress投稿へ進まない。通知はSMTPが使えない可能性があるため、ログと `.eml` バックアップを残し、可能な通知経路だけ試す。

### B. 外部成果物の作成順序を変更する

現在のようにDriveだけ成功してWordPressが失敗する状態を避けるため、最終保存順序を以下にする。

1. 記事生成
2. 品質ゲート
3. アイキャッチ画像ゲート
4. WordPress/SMTPプリフライト
5. WordPress下書き保存
6. WordPress読み返し検証
7. Google Drive確認用テキスト保存
8. 最終通知1通
9. 履歴確定

ただし、WordPress成功後にDrive保存が失敗した場合は、WordPress投稿IDとDrive失敗理由を通知・ログに残す。Drive保存失敗時にWordPress下書きを自動削除するかは、別途仕様判断が必要。

### C. run単位の状態管理を導入する

`westhill` の `RunState` と同じ考え方で、月次runごとに `state.json` を作る。

記録する主なstep:

- `preflight_ok`
- `articles_generated`
- `quality_ok`
- `featured_images_ok`
- `wordpress_posts_created`
- `wordpress_verified`
- `drive_review_texts_saved`
- `notification_sent`
- `processed_pdf_recorded`

再実行時は、既に `wordpress_posts_created` があるrunでは同じ3件を再作成しない。

### D. 通知は once-marker 方式にする

最終通知が送信できた場合だけ、runディレクトリに `notification_sent.json` を保存する。

- markerあり: 再送しない
- markerなし、SMTP復旧済み: 再送可能
- 中間通知: 原則送らない

### E. 本文反復ゲートを追加する

品質ゲートに次を追加する。

- 同一段落の完全一致反復
- 正規化後の類似段落反復
- 同じ文頭パターンの過剰反復
- H2ごとの文章テンプレート化

反復が一定閾値を超えた場合、WordPress保存へ進めない。

### F. 自動実行の完了条件を再定義する

本番採用できる最終検証の合格条件:

- 自動実行が発火する
- 通常フローでPDF号を自律選定する
- 作成済み号を月号順で正しくスキップする
- 3件すべて記事品質OK
- 3件すべて未確認ファクト0
- 3件すべて毎記事新規の写真品質アイキャッチ
- 3件すべてWordPress下書き保存
- 3件すべてWordPress読み返し検証OK
- 3件すべてGoogle Drive確認用テキスト保存
- 最終通知メール1通のみ送信
- 通知本文に、号・掲載箇所・元トピック・WordPress ID・Drive URL・画像品質・重複確認を記載
- 号単位履歴とトピック履歴が確定
- 途中の中間通知、重複投稿、Driveだけ増える状態が発生しない

## 現時点の判断

WordPress投稿と通知メールは、別案件と同じく実現可能。三回目最終検証の失敗を「Codexではできない」と扱うのは誤り。

正しい対応は、成功実装にある次の仕組みをこの案件に移植すること。

- 同一自動実行環境での外部連携プリフライト
- 投稿前停止ゲート
- run単位の状態管理
- 通知once-marker
- 投稿後読み返し検証を完了条件にする設計
- 本文反復ゲート

次の作業は、上記を実装したうえで、最終検証用の自動実行を再度行う。
